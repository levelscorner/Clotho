# Session: 2026-04-07 — Office Hours + Eng Review

## Summary

Ran `/office-hours` (startup mode) followed by `/plan-eng-review` on the Clotho design doc. Produced an APPROVED design doc, identified 22+ engineering issues, and refined the 4-phase plan into a realistic 1A/1B/2/3/4 structure.

## Office Hours Outcomes

### Mode: Startup (pre-product, no users)

### Problem Validated
Solo AI content creators juggle 4-5+ independent AI tools (LLMs for scripts, image generators, video generators, TTS, editing tools) in separate browser windows. The creative pipeline requires constant copy-pasting between tabs. Founder personally tried chaining Gemini gems (Gem1 for script, Gem2 for character, Gem3 for director) and it failed — tedious, error-prone, broke constantly when one gem produced unexpected output.

### Target User Defined
Solo AI content creators with day jobs who produce story-driven or informational video content for YouTube/Instagram/TikTok. They have creative vision and audience psychology instincts but limited time. They already use multiple AI tools and understand prompting.

### Competitive Landscape
- Google Flow: integrated image+video gen in one workspace (closed ecosystem)
- LTX Studio: full production chain in one tool (closed ecosystem)
- ImagineArt Workflow: node-based canvas for content creation (closed ecosystem)
- 80% of creators already use AI in their workflow (Digiday data)
- All major players building closed ecosystems locked to their own models
- Clotho's gap: composable canvas letting creators use PREFERRED tools together

### Five Premises Agreed
1. Core problem is being "human middleware" between 4-5 AI tools
2. Wedge is AI content creators, not enterprise or developers
3. **Revised**: Canvas IS the creative medium. Templates are starting points, not the product. (Founder defended composability against cross-model challenge: "It would be just developers if creating a pipeline is hard. It's not hard. It's like a puzzle piece.")
4. Status quo caps output at ~1 piece/day with burnout risk
5. Enterprise features are year 2-3, not wedge concerns

### Cross-Model Second Opinion
Independent Claude subagent reviewed the session. Key insight: "The Gemini gems failure IS the product spec. LLM chains need typed handoffs, error recovery, and iteration loops — exactly what a DAG engine with port types and cached intermediate results provides."

Challenged Premise #3 (composability). Argued creators want templates, not empty canvas. Founder defended with reasoning. Premise kept with refinement.

### Approach Chosen: "Solid Canvas, Then Everything"
4 phases, 8-12 weeks. Build the complete composable creative canvas first, then layer templates and community features on top. Max effort, product-first.

## Eng Review Outcomes

### Codebase Audit Results
- Backend: ~70% feature-complete for Phase 1
- Frontend: ~80% complete for canvas, 0% test coverage
- Auth: ~50% complete (backend done, frontend unauthenticated)
- 9 backend test files, 0 frontend test files
- Total: ~8,600 lines of application code (not 13K as originally claimed)

### Architecture Issues Found (3)
1. **Streaming gap** — Provider.Stream() exists but engine calls Complete(). Users stare at blank nodes for 30-120 seconds. Elevated to Phase 1A top priority. Requires StepExecutor interface redesign (not just wiring).
2. **Cost cap not enforced** — AgentNodeConfig.CostCap field exists, never checked. 5-line fix in agent_executor.go.
3. **Hardcoded admin password** — Migration 004 seeds "clotho123". Move admin creation to app startup with env var.

### Code Quality Issues Found (2)
1. **RoleConfig.Variables dead field** — Defined in domain/agent.go, never used. Decision: implement variable interpolation in system prompts.
2. **Zero frontend tests** — No *.test.ts files anywhere. Decision: add tests for pipelineStore, historyStore, portCompatibility in Phase 1A.

### Test Coverage
- 9/35 code paths tested (26%)
- Backend: 45% coverage, Frontend: 0%
- 26 gaps identified, 2 critical (crypto round-trip, auth JWT)
- 3 paths need E2E tests, 0 need evals

### Outside Voice Findings (16 issues from independent Claude subagent)
Critical findings:
1. **Streaming is interface redesign** — StepExecutor.Execute() returns single blob, needs channel/callback
2. **Goroutine-per-node conflicts with streaming** — both touch same engine loop, design together
3. **Frontend completely unauthenticated** — api.ts sends no Authorization header, no login page
4. **Refresh token logic bug** — validates expired access token, always returns 401 on normal refresh
5. **Tenant scoping missing** — ExecutionStore.Get() and CredentialStore.Get() take only ID, no tenant check
6. **Map iteration non-deterministic** — concatenateInputs uses Go map, random prompt order
7. **EventBus drops events silently** — 64-item buffer, no backpressure
8. **No cancellation path** — StatusCancelled exists but nothing sets it, no cancel endpoint
9. **Phase 1 overscoped** — 12+ features in 3 weeks for solo developer with day job

### Decisions Made

| Issue | Decision | Rationale |
|-------|----------|-----------|
| Streaming | Phase 1A top priority | #1 UX impact, transforms "watch nothing" into "watch story come alive" |
| Goroutine-per-node | Design both, ship sequential first | Avoids rewriting engine twice. Sequential correct for linear pipelines. |
| Cost cap | Enforce in Phase 1A | 5-line fix, protects solo creators from runaway costs |
| Admin password | Move to env var | Security hygiene before any deployment |
| Variables field | Implement interpolation | Dead code that looks like a feature is worse than no code |
| Frontend tests | Add in Phase 1A | Zero coverage on 392-line state management is a gamble |
| Crypto tests | Add in Phase 1A | Untested crypto protecting API keys is a critical gap |
| Auth tests | Add in Phase 1A | Untested auth is a lockout waiting to happen |
| Refresh token bug | Fix in Phase 1A | Real bug, normal refresh always fails |
| Tenant scoping | Fix in Phase 1A | Correctness issue, simple WHERE clause additions |
| Cancellation | Build in Phase 1A | No stop button + expensive models = cost risk |
| Map iteration | Fix in Phase 1A | 2-line sort fix, deterministic prompts |
| EventBus buffer | Increase + log drops | Prevent silent data loss during streaming |
| Worker poll delay | Keep queue-only | 250ms imperceptible next to 5-30s LLM calls |
| Phase 1 scope | Split into 1A + 1B | 12+ features in 3 weeks unrealistic for solo dev |

### Revised Phase Plan

**Phase 1A — Must-Haves (3 weeks):**
1. Streaming engine refactor (StepExecutor interface redesign)
2. Fix refresh token logic bug
3. Add tenant scoping to Get queries
4. Cost cap enforcement
5. Execution cancellation endpoint
6. Sort map keys in concatenateInputs
7. EventBus buffer increase + drop logging
8. Crypto round-trip tests
9. Auth JWT/password tests
10. Frontend tests (pipelineStore, historyStore, portCompatibility)

**Phase 1B — Should-Haves (2 weeks):**
1. Re-run from any node
2. Error recovery UX
3. API key management UI
4. Frontend auth (login page, token storage)
5. Complete Gemini integration polish

**Phase 2 — Media Integration (3 weeks):**
MediaProvider interface (Submit/Poll), Replicate image gen, Replicate video gen, OpenAI TTS, media preview in nodes, new media node types.

**Phase 3 — Pipeline Export/Import (1 week):**
JSON serialization, export/import, version snapshots.

**Phase 4 — Templates + Landing (3 weeks):**
5 pipeline templates, gallery UI, landing page, auto-layout, keyboard shortcuts.

### Parallelization Strategy (Phase 1A)
- Lane A: Streaming refactor (engine + executors)
- Lane B: Security fixes + tests (auth, crypto, stores)
- Lane C: Frontend tests (stores, portCompatibility)
Launch A + B + C in parallel. Merge all. Then Phase 1B sequential.
